# Tikkle (티끌) : AI 기반 맞춤형 지출 및 식재료 관리 서비스

> **"나도 모르게 새어나가는 작은 지출을 막고, 버려지는 식재료를 구합니다."**
> 단순한 가계부를 넘어, 영수증 OCR과 머신러닝을 활용해 유저의 소비 패턴을 분석하고 최적의 라이프스타일을 제안하는 데이터 기반 프로덕트입니다.

## 프로젝트 개요

* **진행 기간**: 1개월
* **참여 인원**: 4명 (도메인별 분할 및 병렬 개발)
* **담당 직무**: 팀장 (Architecture, AI Pipeline) & 냉장고 도메인 풀스택
* **프로덕트 비전**: 입력의 번거로움을 AI로 없애고, 유저의 행동 데이터 기반으로 실질적인 비용 절감을 제공한다.

## 기술 스택 및 도입 배경

* **Frontend (React Native, Expo)**: 모바일 환경에서의 사용자 행동(카메라 영수증 촬영 등)을 빠르게 검증하고, iOS/Android 크로스 플랫폼 MVP를 최단기간에 배포하기 위해 도입했습니다.
* **Backend (FastAPI, MySQL)**: 4인의 병렬 개발 시 발생할 수 있는 병목을 줄이기 위해, 비동기 처리에 강하고 API 명세서(Swagger)가 자동 생성되는 FastAPI를 채택했습니다.
* **Machine Learning (Random Forest, K-Means)**: 단순 기록형 가계부의 한계를 극복하기 위해 도입했습니다. **Random Forest** 모델로 결측치(영수증에 없는 유통기한)를 자동 예측하여 UX를 개선하고, **K-Means Clustering**으로 유저의 구독 소비 패턴을 군집화하여 데이터 기반의 맞춤형 인사이트를 제공합니다.
* **API & Infra (Vision API, Docker)**: 스마트폰 영수증 촬영 데이터를 텍스트로 변환하기 위해 Google Vision API를 연동했으며, 팀원 간 로컬 환경 불일치로 인한 리소스 낭비를 막기 위해 Docker 컨테이너 기반으로 배포 환경을 통일했습니다.

## 핵심 AI 로직 및 문제 해결 전략

### 1. 냉장고 도메인 : "무엇을 먹을까?" ➔ "어떻게 버리지 않을까?"
단순한 레시피 나열이 아닌, 식재료 폐기율(Loss) 최소화에 집중했습니다.
* **[조합 최적화] 콘텐츠 기반 필터링**: 유통기한이 임박한 '한 종류'의 재료가 아닌, '여러 종류'의 임박한 재료(예: 내일 버려야 할 양파, 3일 남은 두부 등)의 Loss를 합산하고 분석합니다. 이를 통해 임박한 다수의 식재료를 한 번에 소모할 수 있는 최적의 메뉴를 매칭률과 함께 산출합니다.
* **[UX 한계 돌파] OCR + ML 유통기한 자동 기입 파이프라인**: 영수증에는 '유통기한'이 찍히지 않아 유저가 일일이 날짜를 기입해야 하는 치명적인 UX 결함이 있었습니다. 이를 해결하기 위해 OCR로 추출된 품목 데이터를 **랜덤 포레스트** 예측 모델과 연동, 카테고리별 부패 주기를 학습한 시스템이 영수증 인식 즉시 적절한 유통기한을 계산하여 자동 기입하도록 구현했습니다.

### 2. 구독 도메인 : "나와 비슷한 사람들은 어떻게 지출을 관리할까?"
지불 중인 비용은 낮추고, 누릴 수 있는 혜택은 극대화하는 것에 초점을 맞췄습니다.
* **[구독 MBTI] K-Means Clustering**: 유저가 입력한 초기 데이터(연령대, 관심사)와 구독 중인 서비스 결제 금액을 바탕으로 군집화를 진행합니다. 이를 통해 유저를 특정 페르소나(예: OTT 헤비 유저, 알뜰 쇼퍼)로 분류하고, 해당 그룹 내 만족도가 가장 높은 '표준 조합'을 추출합니다. *"OO님과 같은 30대 직장인의 85%는 쿠팡 하나로 통합하여 월 4,900원을 아끼고 있습니다"*와 같이 데이터에 근거한 타겟팅 액션을 유도합니다.

## 내 역할 (팀장 & 냉장고 도메인 전담)

프로젝트 총괄 리더로서 AI 핵심 솔루션 기획과 공통 아키텍처 설계를 주도했으며, 냉장고 도메인의 문제를 풀스택(FE/BE/AI)으로 직접 해결했습니다.

* **프로덕트 기획 및 AI 솔루션 설계 (PM)**: '유통기한 자동 예측'과 '구독 군집화 분석' 등 이 서비스의 심장이 되는 핵심 AI 기능들의 아이디어링을 주도했습니다. 사용자의 불편함을 찾아내고, 이를 어떤 머신러닝 알고리즘으로 해결할 수 있을지 기술적 활용 방안을 팀원들에게 제시하고 기획을 리드했습니다.
* **확장 가능한 아키텍처 및 시스템 설계(Architecture)**: 4명이 도메인별로 병렬 개발하는 환경에서 병합 충돌을 최소화하기 위해 **코드 간 결합도를 낮춘 독립적인 도메인 중심 아키텍처**를 설계했습니다. 
  - **진입점 분리**: 새로운 기능이 추가될 때마다 `App.js`나 `main.py` 같은 메인 진입점을 직접 수정할 필요가 없도록 라우터를 구조화했습니다.
  - **공통 환경 설정 (Config)**: API 엔드포인트를 하드코딩하지 않고 '공통 API 클라이언트 및 환경 설정 파일'로 분리하여, 단 한 줄의 코드 변경만으로 로컬 개발 환경과 실서버 배포 환경을 전환할 수 있도록 편의성과 안정성을 극대화했습니다.
* **데이터 정규화 파이프라인 구축**: 영수증에서 추출된 비정형 텍스트 데이터를 시스템이 인식할 수 있는 카테고리로 정제 및 매핑하여, 통계 화면과 냉장고 DB에 즉시 렌더링되도록 구현했습니다.
* **냉장고 도메인 풀스택 개발**: React Native를 활용한 카메라 뷰 및 인터페이스 UI 구현부터 FastAPI 라우터 설계, DB 쿼리 최적화까지 냉장고 파트의 End-to-End 개발을 전담했습니다.

## 프로젝트 마일스톤 및 개발 로그

단순한 기능 구현을 넘어, 4주간의 체계적인 스프린트 목표를 설정하고 사용자 경험과 데이터 신뢰도를 높이는 방향으로 프로덕트를 발전시켰습니다.

### Sprint 4 & Release (3/30 - 4/10) : 정식 MVP 릴리즈 및 서비스 통합
개별 개발된 냉장고/구독 도메인을 하나의 플랫폼으로 완벽히 융합하고 통합 QA를 거쳐 AWS배포했습니다.
- **도메인 융합 및 UX 통합**: 로그인 유저 ID를 기반으로 냉장고 데이터와 구독 지출 현황이 메인 홈 대시보드에 실시간으로 교차 반영되도록 렌더링 로직을 최적화했습니다.
- **트러블슈팅 및 안정성 확보**: 도메인 간 데이터 통합 과정에서 발생한 통계 페이지 참조 에러 등을 디버깅하고, 모바일 기기별 UI 해상도 이슈를 수정하여 최종 안정화를 마쳤습니다.

### Sprint 3 (3/23 - 3/27) : 핵심 비즈니스 로직(AI/데이터) 연동 완료
확정된 데이터 규격을 바탕으로 실질적인 가치를 제공하는 백엔드/AI 로직을 집중 결합했습니다.
- **OCR 자동화 달성**: [카메라 촬영 ➔ Vision API 텍스트 추출 ➔ DB 자동 저장]으로 이어지는 영수증 파이프라인을 완성하여 수동 입력의 번거로움을 완전히 해결했습니다.
- **ML 예측 로직 적용**: **Random Forest** 기반 유통기한 예측 로직을 결합하여 영수증 인식 즉시 개인화된 유통기한이 자동 기입되도록 구현했습니다.

### Sprint 2 (3/16 - 3/20) : 프론트엔드 전체 UI 선행 개발 및 전체 DB 스키마 설계
화면 구현을 선행하여 데이터의 흐름을 정의하고, 이를 바탕으로 전체 데이터베이스 구조를 확립했습니다.
- **전체 UI 뷰(View) 선행 개발 및 규격 확정**: 백엔드 개발 전, 앱 전체의 프론트엔드 화면과 네비게이션 플로우를 선제적으로 모두 구현했습니다. '껍데기'를 먼저 완성해 봄으로써 각 화면 렌더링에 실질적으로 필요한 데이터 필드(Key)를 누락 없이 도출하고 명확한 API 규격을 확정했습니다.
- **전체 DB 스키마 통합 설계**: 각 도메인[냉장고, 구독, 공통(유저 등)]에서 요구되는 데이터 속성들을 취합하고, 도메인 간의 관계와 확장성을 고려하여 전체 스키마 구조를 설계했습니다.

### Sprint 1 (3/9 - 3/13) : 기획 및 초기 환경 세팅
팀 전체의 설계도를 완성하고 인프라 기초 공사를 마쳤습니다.
- **환경 통일**: 팀원 간 로컬 환경 불일치를 막기 위한 Docker 컨테이너 세팅 및 GitHub Git-flow 브랜치 전략을 수립했습니다.
- **설계 완료**: Figma를 활용한 와이어프레임 설계 및 공통 유저 인증(JWT)을 위한 DB 모델링 초안을 완성했습니다.

## 프로젝트 회고 및 빌더로서의 성장

> **"문제의 근본 원인을 파악하고 시스템으로 해결하는 설계자"**

이전 프로젝트(메추리)에서 겪었던 잦은 버전 충돌과 소통의 병목은 저에게 뼈아픈 교훈이었습니다. 이를 거름 삼아 이번 티끌 프로젝트에서는 무작정 코딩을 시작하는 대신, **'문제 재발 방지를 위한 개발 환경과 시스템 구축'**에 전보다 많은 시간을 투자했습니다.

도커로 팀원들의 개발 환경을 통일하고, 새로운 기능이 추가될 때마다 메인 진입점(`App.js`, `main.py`)에서 충돌이 나지 않도록 라우터 결합도를 낮추는 작업이 그 시작이었습니다.
사실 처음부터 '프로덕트 빌더'라는 거창한 목표를 안고 시작한 프로젝트는 아니었습니다. 하지만 개발 과정에서 마주한 병목 현상들을 시스템으로 타파해 나가는 과정에서 깊은 흥미와 성취감을 느꼈습니다. 이번 티끌 프로젝트는 단순히 새로운 기술 스택을 써본 것을 넘어, 협업 시 발생하는 문제의 원인을 찾아내고 이를 시스템으로 해결해 내는 저의 적성을 발견하게 해 주었으며, 앞으로 **'문제 해결형 프로덕트 빌더'**로 성장하겠다는 명확한 방향성과 확신을 갖게 해 준 가장 값진 경험이 되었습니다.
